# 8.1 第二次请求 - 缓存问题

## 浏览器 HTTP 缓存主要是按 URL（资源地址）存的，不是 IP

浏览器检查缓存时，核心依据：URL

包括：协议 + 域名 + 端口 + 路径 + 查询参数

即：https://example.com:443/js/app.js?v=1

## 第二次点击按钮时

真正“决定要不要发请求”的：是浏览器， 准确说：浏览器网络模块（network stack）在检查缓存。

- 浏览器内部大概结构

  ```bash
    JS代码
      ↓
    fetch/axios
      ↓
    浏览器 Web API
      ↓
    浏览器网络模块
      ↓
    缓存检查
      ↓
    决定是否联网
  ```

## 第二次请求，整体流程（核心）

```bash
  浏览器请求资源
      ↓
  检查强缓存
      ↓
  命中？
   ┌───────┴───────┐
   是               否
   ↓                ↓
  直接使用缓存      发请求到服务器
                    ↓
                协商缓存检查
                    ↓
              资源没变？
             ┌──────┴──────┐
             是             否
             ↓              ↓
          返回304        返回200新资源
```

### 第一层：强缓存（最重要）

强缓存：根本不会发网络请求, 直接读磁盘/内存缓存;

浏览器怎么知道能不能直接用？靠响应头;

```bash
   # 1. Cache-Control（现代主流）
   Cache-Control: max-age=3600 # 表示：3600秒内直接用本地缓存

   # 2. Expires（老方案）
   Expires: Wed, 21 Oct 2026 07:28:00 GMT # 表示：这个时间前都不用请求

```

### memory cache（内存缓存） 🆚 disk cache（磁盘缓存）

- memory cache（内存缓存）：资源在：浏览器进程占用的电脑内存（RAM）里。

  比如：JS、CSS、小图片 刷新页面可能还在。
  但：关闭标签页可能消失

- disk cache（磁盘缓存）：存在硬盘。

  即使关闭浏览器：可能还在，速度比内存慢一点。

### 第二层：协商缓存（重要）

服务端第一次返回：`ETag: "abc123"`或`Last-Modified: Tue, 01 May 2026 12:00:00 GMT`浏览器记住。

第二次请求时,浏览器会带上：

```bash
   # 1. ETag方案（主流）:更准确，通常：文件hash、版本号
   If-None-Match: "abc123" # 表示：我本地版本是 abc123, 你看看变了没？

   # 2. Last-Modified方案: 精度是秒：1秒内修改两次可能检测不到
   If-Modified-Since: Tue, 01 May 2026 12:00:00 GMT # 表示：这个时间后改过吗？

```

- 如果：资源没变, 返回：304 Not Modified, 📌注意：没有响应体

  因为：浏览器本地已经有资源, 浏览器：直接使用本地缓存

- 如果资源变了, 服务器返回：200 OK, 并返回：新资源, 浏览器更新缓存。

### 为什么协商缓存还要发请求？

因为：浏览器不确定资源有没有更新，只能问服务器。
